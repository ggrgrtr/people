# __Работа начинается в `main(..)`__ 

Запускаем программу: `start.py` -> `app.py->main(..)`

Конфиг - объект предсозданного класса `AppConfig()`. Его будем передавать в другие классы и функции, чтобы испольщовать в их работе выбранные заранее параметры, которые все сосредоточены в `AppConfig()`

Создаем объект считывания аргументов командной строки с `args = parse_args()` для автоматического выбора параметров работы при запуске

Переходим при возможности на `GPU` и оптимизируем вычисления с `torch.backends.cudnn.benchmark = True`
Это позволит задействовать большую мощность для обработки детекции и признаков человека


__Создаем ряд объектов:__
- Создаем объект `YOLO`-детектор людей класса `PersonDetector`
- Создаем объект для извлечения признаков `ReID-embending` и цветовых гистограмм детекции в классе `AppearanceEncoder`
- Создаем объект для получения `embending` лица класса `OptionalFaceBackend`
- Создаем объект-трекер класса `TrackletTracker`. Это список красткосрочных треков, хранящих свои признаки и подтверджающиеся поинтервально детекцией
- Создаем объект для сохранения трека в долгосрочную память `identity_manager = IdentityManager(config)`. Именно `IdentityManager` присваивает постоянные `person_id`, сравнивая `ReID`, цвет, форму и лицо. Это происходит через вызов
- Создаем пути и папки для сохранения результата через `build_output_paths()`

Трекер `TrackletTracker` ведет список коротковерменных треков и проверяет, задетектирован новый трек или это продолжение одного из текущих. `IdentityManager` сопоставляет признаки надежных треков с долговременными личностями для присвоения `person_id` или создания нового.
Без `IdentityManager` трекер может оборваться и создавать новые треки для тех же людей
`IdentityManager` хранит словарь `identities`. Каждый элемент словаря это объект `IdentityRecord`, который хранит и обновляет долговременные характеристики человека, получаемые из его детекции и последующего трека.

### ==Чтение кадра==
С `ThreadedCameraCapture` происходит захват изображения с библиотекой `cv2` и его передача в фоновый поток `threaded_capture`, основанный на библиотеке `threading`. Пока фоновый поток считывает изображение с видео, основной поток может проводить вычисления, детекции по предыдущему кадру. 
Потоковый захват работает только для live-камеры. Для видео `threaded_capture is None` и запись идет прямо через `cv2` в `cap.read()`.

Если основной поток провел все вычисления, то он входит в спящий режим, дожидаяь нового кадра от `threaded_capture` для продолжения работы. Это требуется, чтобы `main()` цикле постоянно не отправлял запрос на получение нового кадра и не создавал __`busy waiting`__
1. Определяем источник видео через `source, source_label = resolve_source(args.source)`
2. Создаем объект считывания кадра в `cap = open_capture(source, config)` 
3. Создаем фоновый поток
	1. `threaded_capture = ThreadedCameraCapture(cap).start_flow()`
	2. В инициализации класса создается синхронизация потоков, а затем и сам фоновый поток
4. Уже в цикле ==`while True`== запускается `threaded_capture.read2()`. Если недообработан предыдущий кадр, ожидание не мешает выполнять другие части программы
	1. Цикл в функции проверяет, если текущий кадр в `main()` соответствует кадру из фонового потока по своему номеру, то `read2()` с `condition` усыпляет основной поток на 1 сек.  
	2. Если кадры не соответствуют, то функция возвращает кадр `self.latest_frame`, полученный классом при фоновом выполнении `_reader()`, вида `(height, width, 3)` с 3 слоями `BGR` в формате `numpy.ndarray`
5. Так `main()` получает новый кадр-`frame` через объект класса фонового потока

Определяем флаг `should_detect` и проверяем условие `should_detect`, что пришло время детекции

### __Если время детекции__

### ==`YOLO-detection:`==
Если время детекции `if should_detect`, то детектируем людей с `detections = detector.detect(frame)` и получаем список кортежей рамок детекций вида `(x, y, w, h)`

При создании объекта `detector` в файлах программы ищется архитектура нейросетей `YOLO` для работы с `ultralytics` 
Детектор загружает модель `yolov8n.pt` из отдельного предустановленного файла. Перед передачей в модель, кадр сжимается, чтобы ускорить обработку изображения. При этом появляются потери в точности детекции, в особенности мелких объектов. Это понижает собственную уверенность детекции. 
Функция `detect(..)` получает кадр и с помощью `YOLO` находит детекции, описывает для них рамку с определенной уверенностью и обрезает по кадру. Далее список детекций объединяется с уверенностью как список кортежей `((x, y, w, h), confidence)`
Список передается в `self._suppress_duplicate_detections()` и очищается от дублей, опираясь на коэффициент `confidence`, `IoU`, пересечение площадей и вложенности рамок. После возвращается в `app.py` как список из `(x, y, w, h)` каждой отдельной рамки детекции для дальнейшего подтверждения треков или создания новых и извлечения признаков.


#### Обработка и извлечение `ReID:`
Исходя из детекций определяем, для кого считать затратный `ReID-embending`:

 Создаем `candidate_feature_indices` - список индексов кандидатов детекций на вычисление признаков, получаемый из рамок детекций, плохо сопоставимых с активными треками. 
_Передаем список кортежей рамок `(x, y, w, h)` детекций и возвращаем список индексов списока подходящих детекций_
```
tracker.reid_candidate_detection_indices(
                detections,
                frame.shape)
```
Это требуется, чтобы вычислить признаки для детекций, у которых нет хорошо совпадающих текущих треков.
Благодаря полученным признакам детекций будет проще найти для детекций трек или создать новый, имеющий признаки для повторного сопоставления.
Чтобы добавить трек в долгосрочный поиск, он должен накопить достаточно признаков, поэтому вычисляем `ReID`.

Для каждого трека при подсчете `ReID` растет `self.feature_updates`. При достижении `self.config.identity_min_feature_updates`, доверенном треке, достаточных `identity_min_hits` и площади рамки, трек переходит в долговременную память.
Поэтому, если трек надежный, активно считается `ReID`, чтобы добавить трек в долговременную память и дать личный `person_id`, по которому трек будет связываться с конкретным человеком

Это позволит запоминать людей и хранить их `person_id`, даже если завершится их текущий трек


Далее создаются флаги для определения того, по какой схеме потом будут определяться принаки
- Флаг `has_pending_identities` ставится, если есть надежные треки, не вошедшие еще в _долговременную память_ и не имеющие `person_id`
- Флаг `need_full_reid` проверяет, нужно ли проводить выявление признаков для всех детекций
- Флаг `should_extract_features` определяет, нужно ли вообще вычисление признаков

Создаем список индексов рамок для вычисления признаков
__`selected_feature_indices = None if should_extract_features else []`__

- Если `include_features=False`, то не считаем `ReID` вообще
- И, если нужно вычислять `ReID` `include_features=should_extract_features`, получаем в __`selected_feature_indices`__ индексы из `candidate_feature_indices`
- Если нет подходящих элементов, `selected_feature_indices` получает через сортировку индексы самых больших рамок детекций. Чем больше рамка, тем увереннее в ее детекции мы можем быть. Таким образом мы вычисляем признаки для новых детекций, еще не имеющих треков
- Если `need_full_reid==True`, то `selected_feature_indices=None`. Тогда в `encoder.extract(..)` при заданном изначально `max_feature_boxes=None` считаются признаки для всех детекций, которые в` tracker.update(..`) либо сопоставятся с текущими треками, либо привяжутся к ново созданным, которые в свою очередь при следующей детект-итерации по этим признакам будут связываться с новой детекцией и ее признаками
Если нужен полный `ReID`, то `selected_feature_indices=None`, то получаем спиок из всех индексов рамок детекций


Такой способ работы с `ReID` экономит ресурсы системы, повышает `fps` и эффективность других вычислений


##### __Вычисляем признаки и цветовые гистораммы для выбранных кандидатов__

Извлекаем признаки лица, передаем индексы в `selected_feature_indices`
```
face_features = face_backend.extract_batch(
                frame,
                detections,
                candidate_indices=selected_feature_indices)
```

```
features, color_histograms = encoder.extract(
                frame,
                detections,
                include_features=should_extract_features,
                max_feature_boxes=None,
                feature_indices=selected_feature_indices)
```
`features, color_histograms` - признаки и цветовые гистограммы
Цветовые гистограммы вычисляются всегда, не зависимо от условий для эмбендинга, из-за дешевезны операции

Передаем флаг `should_extract_features`, что нужно вычислить признаки эмбендинг и передаем список индексов рамок детекций `selected_feature_indices` на вычисление из  списка кортежей всех рамок детекций detections. По списку `selected_feature_indices`, переданному в `extract(..)`, вычисляем `ReID`. Если список пустой, вычисляем признаки для всех детекций.

###### __`Цикл по всем кортежам boxes`__
Получаем из кортежа рамки __`crop`__, хранящий массив значений изображения кадра `frame`.

По всем детекциям строим цветовые гистограммы в `_build_color_descriptor(..)`.
- Переводим кроп из `BGR` в `HSV` - оттенок, насыщенность, яркость. 
- Разделяем каждый канал на зоны, получаем цветовые гистограммы, учитывающие общую цветовую зону и по секторам, чтобы более точно различать людей по цвету. 
- Добавляем среднюю яркость и насыщенность, нормализуем, склеиваем и возвращаем признак в виде массива `3x98` значений

Если индекс кортежа рамки совпадает с индексом для `ReID`, то текущий кроп преобразуем в формат, который принимает модель
- из `BGR` в `RGB`, тк. модель `ResNet50` обучалась на датасете `ImageNet`, находившемся в формате `RGB`.
- Изменяем размер массива к `224x112`, какой вход принимает модели
- Перемещаем `[H,W,C] -> [C,H,W]`  и нормализуем делением на 255
- `ImageNet`-нормализация и преборазование в тензор

###### После цикла
Объединяем тензоры в `batch`.
Передаем модели `self.model()` данные и получаем `embeddings (n, 8192)` и нормализуем. Потом эти признаки передаются в трекер `TrackletTracker.update(..)`, и уже там попадают в нужные треки


### ==`ReID-Model`==

Для вычисления признаков класс `AppearanceEncoder()` поднимает свою модель в виде объекта класса `ResNet50ReIDBackbone()`, который в свою очередь настраивает нейросеть `ResNet50` с `ImageNet`-весами функцией `_build_model(..)`. 
`ResNet50` в меру сильная и быстрая, что дает хороший баланс.

От изначальной модели `torchvision.models.resnet50` берется только сверточная часть `backbone`. Классификационная часть (последняя) для предсказания убирается и заменяется на адаптивный и `GeM` пулинги. Таким образом модель класса ничего не предсказывает и лишь возвращает `embedding` человека-детекции, который будет использоваться для вычленения признаков

`GeM pooling` усиливает более выделяющиеся значения тензора и подавляет слабые, что позволяет сфокусироваться на более ярких и выделяющихся признаках
Адаптивный пулинг разбивает анализ кропа на зоны, что дает большую устойчивость признака, и совмещается с общим `GeM` пулингом. Получается конечный нормализованный `embedding`.
Используем вытянутый вверх тензор `112x224`, опираясь на то, что камера будет фиксировать скорее стоящих или сидящих людей.

Далее в `utils.resolve_reid_weights(..)` происходит поиск в папке программы пути к файлу `.pth`, содержащему настроенные слои модели архитектуры для `ReID`. Внутри программа фильтрует и очищает `checkpoint`, обрезая префиксы и переименовывая словарь весов к виду блока слоев `self.stem` в соответствии со структурой модели, далее загружая все совпавшие слои. В данном случае подгружает эти словари весов файла в первые слои модели `self.stem.0` и `self.stem.1`. 
Ставим модель в режим валидации для фиксации весов и получения только выхода. `Batch`-нормализация и другие слои работают стабильно для инференса

Из рамки вырезается `crop` детекции или активного трека с падингом вокруг `bbox`, чтобы захватить больше контекста для лучшего распознавания, и обрабатываем случаи, когда кроп выходит за границы изображения, что обеспечивает более устойчивое извлечение признаков даже при неточных `bbox`
Модель получает кроп человека и возвращает нормализованный `8192`-мерный `ReID`-вектор для сопоставления между кадрами и с долгосрочной памятью.

__Вход__ `float32`
- `(n, 3, 224, 112)`
	- `n` -количество кропов
	- 3 - RGB
	- 112 - ширина
	- 224 - высота
- После производной функции `forward()`
	`(n, 2048, 7, 4)`
	- `n` -количество feature map
	- 2048 каналов признаков
	- 7x4 - пространственная сетка

__Для глобального признака__
- `GeMPooling2d` + вытягивание
	`(n, 2048, 7, 4) -> (n, 2048, 1, 1) -> (n, 2048)`

__Частичные признаки__
- `AdaptiveAvgPool2d` + `squeeze(-1)` + `transpose(1, 2)`
	`(n, 2048, 7, 4) -> (n, 2048, 3, 1) -> (n, 2048, 3) -> (n, 3, 2048)`
- Нормализация и `reshape`
	`(n, 3, 2048) -> (n, 6144)`

__Объединение признаков и выход__
	`(n, 8192)`


#### Обновляем `tracker:`
Полученные признаки
- `features`
- `color_histograms`
- `face_features`
передаем в `update(..)` и проходимся по списку треков, предсказываем Калман. 
Сравниваем детекции и текущий список треков с `_associate_tracklets()` и получаем `matches` - совпадения, `unmatched_tracks` - треки без детекции и `unmatched_detections` - детекции без треков.
Здесь мы 
- В matches обновляем активные треки, подтвержденные детекцией  с `for track_index, det_index in matches:`.  В каждом треке обновляем и сглаживаем рамку по детекции, корректируем предсказание Калмана, обновляем признаки внешности и форму рамки. Строим траекторию движения

- В `unmatched_tracks` штрафуем треки, если они не зафиксированы `YOLO` с `track.mark_missed`. И завершаем, если штраф привысил лимит, добавляя в `finished_tracklets`. В ином случае сохраняем в `still_active`. Обновляем список активных треков с `self.active_tracklets = still_active`

- В `unmatched_detections` создаем новые треки из детекций `new_track = Tracklet(..)` и обновляем список активных треков с `self.active_tracklets.append(new_track)`


Активные треки - список краткосрочной памяти, анализирующий и прикрепляющийся к человеку только в моменте
Очищаем список завершенных треков от дублей

Все массивы `features`, `color_histograms`, `face_features` идут по тем же индексам, что и `detections`. То есть `features[3]` относится к `detections[3]`.

предсказанное положение трек с новой детекцией
`finished_tracklets = tracker.predict_only(frame.shape)`


Завершаем долговременные треки из списка завершенных треков в историю видео
`identity_manager.finalize_tracklets(finished_tracklets)`


`observed_tracklets = [track for track in visible_tracklets if track.was_observed()]`
Получаем список наблюдаемых в текущем кадре треков `tracklet`  и передаем его в

`identity_manager.observe_tracklets(observed_tracklets, ...)`
Функция анализирует доверенные треки, имеющие текущую детекцию.
Если у треков достаточно накопленной информации, то объединяет их с долговременным объектами `IdentityRecord()`, хранящими признаки и особенности личности, записывает сводку о личности в память с `JSON`
Накопленные признаки доверенного трека сравниваются с признаками долговременной личности, после сопоставления треку дается существующий `person_id` этой личности
При несповпадении создает новую `identity`-личность `IdentityRecord()` и выдаем треку новый `person_id`. Объект `IdentityRecord` хранит накопленный портрет человека на протяжении всей сессии

Далее `identity.record_observation` обновляет долговременные признаки личности - смешиваем старые и новые. Это позволяет сохранить признак устойчивым во времени

`PersonDetector` находит `bbox` людей → `AppearanceEncoder.extract()` считает признаки по этим `bbox` → `TrackletTracker`  используют признаки, чтобы создавать треки `Tracklet` и сопоставлять людей между кадрами → передает надежный трек в долгосрочной памятью `IdentityManager`


`detector.py`:

проверка на дублирующиеся и шумовые детекции

`reid.py`:
энкодер
объект класса `ResNet50ReIDBackBone` импортирует `ResNet50` 4 слоя с `ImageNet`-весами, замена пулинга на `GEM`
объект класса `ApperienceEncoder()` создает бэкбон модель, считывает из файла `ReID`-веса, форматирует их и подгружает в `backbone` для более глубокого считывания признаков. Создание и обработка цветового дескриптора. возвращаем эмбендинг свойств кропа
